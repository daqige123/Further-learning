# SSE和websoket

SSE 是基于 HTTP 的服务器推送协议，只支持服务器向客户端单向发送数据，浏览器通过 EventSource API 接收消息，并且支持自动重连和断点续传。

WebSocket 是一种独立的全双工通信协议，通过 HTTP Upgrade 建立连接后可以实现客户端和服务器之间的双向通信，并且支持文本和二进制数据。

在实际应用中，如果只是服务器单向推送，比如 AI 流式返回或日志流，可以使用 SSE；如果需要实时双向通信，比如聊天或协作系统，则更适合使用 WebSocket。

| 对比维度    | WebSocket                                                                  | SSE (Server-Sent Events)                                                                               | 面试加分项解读                                                                                                                   |
| :---------- | :------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------- |
| 1. 通信方向 | 双向 (Full-duplex)``客户端 ↔ 服务器                                       | 单向 (Simplex)``服务器 → 客户端                                                                       | 关键点：如果业务需要客户端频繁上报数据（如游戏操作、即时聊天），必须选 WebSocket。如果只是展示数据（如大屏、通知），SSE 更简单。 |
| 2. 协议基础 | 独立协议 (`ws://`/`wss://`)``需要特殊的握手升级 (Upgrade: websocket)。 | 基于 HTTP (`http://`/`https://`)``本质是特殊的 HTTP 长连接 (`Content-Type: text/event-stream`)。 | 关键点：SSE 天然兼容 HTTP 生态（Cookie、Header、认证），防火墙穿透性好；WebSocket 需要服务器专门支持协议升级。                   |
| 3. 数据类型 | 二进制 & 文本``支持发送 Blob, ArrayBuffer 等二进制数据。                   | 仅文本``默认只支持 UTF-8 文本（虽然可以传 JSON，但不能直接传二进制）。                                 | 关键点：涉及文件传输、音视频流、高频二进制包（游戏帧），WebSocket 是唯一选择。                                                   |
| 4. 连接管理 | 手动维护``需自己写心跳检测 (Ping/Pong)、断线重连逻辑。                     | 浏览器原生支持``自动处理断线重连 (`reconnect`字段)，自动心跳。                                       | 关键点：SSE 开发成本极低，几行代码就能实现稳定的重连；WebSocket 需要引入库或写大量样板代码来保证稳定性。                         |
| 5. 兼容性   | 极好``所有现代浏览器及 Node.js/Java/Go 后端均支持。                        | 较好 (但有短板)``现代浏览器支持良好，但 IE 不支持，且部分旧安卓 WebView 有坑。                         | 关键点：如果必须支持 IE 或极老旧环境，只能用 WebSocket 或轮询。                                                                  |

### SSE 自动重连和断点续传

1. 大多数现代浏览器支持Sse 自动重连，但是可能在网络不好的情况下增加请求风暴，所以建议增加指数退避。
2. **实现断点续传需要客户端和服务端协同工作：服务端****：为每一条推送的消息分配一个唯一的、递增的**`id`。**文本****编辑**

```
   1id: 1001
   2data: {"msg": "第一条消息"}
   3
   4id: 1002
   5data: {"msg": "第二条消息"}
```

**客户端** **：**

* **在接收到消息时，从** `event` **对象中获取** `lastEventId` **并持久化存储（例如存储在变量或** `localStorage` **中）。**
* **在发起重连时，将最后收到的** `id` **通过查询参数（如** `?lastId=1001`）或请求头（`Last-Event-ID: 1001`）发送给服务端。

**服务端（续）** **：接收到带有** `lastId` **的请求后，从消息队列或数据库中查找该** `id` **之后的所有消息，并推送给客户端，从而实现续传。**
